# requirements.md

## 1. 文档目标

本文档定义 `attention-pqc` 代码库新增 **基于 MindQuantum 的 VQE 求解能力** 与 **基于 MindSpore 的模型构建能力** 的需求范围、目标、约束、验收标准与边界，为后续技术设计与任务拆分提供唯一需求基线。

## 2. 项目背景

当前代码库已经具备两条相对独立的能力链路：

1. **Julia + Yao.jl 的 VQE 数据生成链路**：用于定义参数化量子线路、构造 cluster-Ising 类哈密顿量、执行 VQE 优化并导出参数数据集。
2. **Python + PyTorch 的模型训练链路**：用于读取量子数据集并训练 VAE / Attention VAE / Diffusion 等经典模型。

本次新增能力的目标不是替换现有全部实现，而是在当前仓库中补充一条 **Python 原生、可扩展、可验证** 的新能力链路：

- 使用 **MindQuantum** 承载量子线路、哈密顿量与 VQE 求解；
- 使用 **MindSpore** 承载模型封装、训练接口与后续可扩展的量子-经典混合建模能力。

## 3. 需求范围

### 3.1 In Scope

本次需求覆盖以下内容：

1. 在当前仓库中新增一套 **Python 实现的 VQE 求解模块**；
2. 该求解模块使用 **MindQuantum** 完成：
   - 参数化量子线路构建；
   - 哈密顿量表达；
   - 期望值与梯度计算；
   - VQE 参数优化；
3. 使用 **MindSpore** 对新增求解过程中的模型/训练单元进行封装；
4. 支持对当前仓库现有问题设置进行复现或等价迁移，包括：
   - 层状参数化线路；
   - cluster-Ising / transverse-Ising 类自旋哈密顿量；
   - 参数扫描式 VQE 求解；
5. 支持将求解结果以当前仓库可消费的结构导出，用于后续数据分析或模型训练；
6. 为后续将 MindSpore 模型扩展到现有生成模型链路提供可演进基础。

### 3.2 Out of Scope

本次需求明确不包含以下内容：

1. 完全删除、重写或替换现有 Julia / Yao.jl 实现；
2. 完全删除、重写或替换现有 PyTorch 模型实现；
3. 在本阶段完成所有现有 PyTorch 模型到 MindSpore 的一比一迁移；
4. 引入分子化学专用 UCCSD 工作流作为主线能力；
5. 构建分布式训练、Web 服务、可视化前端或完整实验平台；
6. 对现有 notebook 做大规模重构，除非设计阶段认定为最小必要改动。

## 4. 问题陈述

当前代码库存在如下限制：

1. **VQE 主链路不在 Python 中**，导致与现有 Python 数据处理和模型实验链路存在语言割裂；
2. **量子计算栈与模型栈不统一**，当前量子部分基于 Julia/Yao，模型部分基于 PyTorch，缺少统一的 Python 生态入口；
3. **缺少 MindQuantum / MindSpore 支持**，无法基于 MindSpore 生态继续扩展混合量子-经典训练流程；
4. **当前实验流程复用性有限**，主要以脚本与 notebook 驱动，缺少明确的 Python 模块化求解边界。

## 5. 目标

### 5.1 业务/研究目标

1. 为仓库提供一条 **可复现、可维护、可扩展** 的 MindQuantum VQE 求解链路；
2. 使该链路可与现有 Python 数据处理流程自然衔接；
3. 为未来基于 MindSpore 的量子-经典混合模型研究提供统一基础设施；
4. 降低对 Julia 环境的依赖，使核心求解实验能够在 Python 环境中完成。

### 5.2 工程目标

1. 新功能应以 **独立模块** 方式接入，不破坏现有代码结构；
2. 新功能应具备明确的输入、输出、配置与运行边界；
3. 新功能应能在小规模问题上与现有实现或理论结果进行一致性验证；
4. 新功能应为后续扩展更多哈密顿量、线路模板和训练策略保留接口。

## 6. 角色与使用场景

### 6.1 目标用户

1. 研究自旋系统与 VQE 的算法研究者；
2. 希望在 Python 环境下统一完成量子求解与模型构建的开发者；
3. 需要基于导出参数数据继续训练生成模型的实验人员。

### 6.2 核心使用场景

#### 场景 A：单次 VQE 求解
用户指定系统规模、线路深度、哈密顿量类型和优化配置，运行一次 MindQuantum VQE，得到：
- 最优能量；
- 最优参数；
- 收敛过程；
- 可选的基态参考对照信息。

#### 场景 B：参数扫描求解
用户对一个控制参数区间进行扫描，系统逐点执行 VQE，并支持以上一参数点结果作为下一点初始值，输出一组连续结果。

#### 场景 C：结果导出并供下游训练
用户将每个参数点的最优线路参数、标签和值导出为结构化数据，以复用现有数据读取与训练流程。

#### 场景 D：后续混合模型扩展
开发者在 MindSpore 中复用量子求解/量子层封装能力，将其嵌入新的模型或训练实验。

## 7. 功能需求

### 7.1 求解器能力

#### FR-1 Python 原生 VQE 能力
系统必须提供 Python 原生 VQE 求解能力，且核心量子求解逻辑基于 MindQuantum 实现。

#### FR-2 参数化量子线路构建
系统必须支持构建与现有仓库问题域相匹配的参数化量子线路，至少覆盖：
1. 单比特旋转层；
2. 双比特纠缠层；
3. 多层堆叠 ansatz；
4. 清晰、稳定的参数命名和参数顺序。

#### FR-3 哈密顿量表达
系统必须支持用 MindQuantum 表达当前仓库所需的自旋哈密顿量，至少包括：
1. transverse-Ising 类项；
2. cluster-Ising 类项；
3. 周期边界条件相关项；
4. 可组合的 Pauli 串表达方式。

#### FR-4 能量与梯度计算
系统必须支持对给定线路和哈密顿量计算：
1. 期望值能量；
2. 相对于可训练参数的梯度；
3. 在优化循环中可重复调用的稳定接口。

#### FR-5 优化循环
系统必须支持完成完整 VQE 优化流程，包括：
1. 参数初始化；
2. 迭代更新；
3. 收敛判定；
4. 记录最优参数与最优能量；
5. 输出训练/优化历史。

#### FR-6 Warm Start / Continuation
系统必须支持参数扫描场景下的 warm start，即：
- 当前参数点的最优参数可作为下一参数点的初始化输入。

#### FR-7 基准对照
系统必须支持在小规模问题上计算或接入基准结果，用于比较：
1. VQE 最优能量；
2. 收敛误差；
3. 可选的参考基态能量。

### 7.2 MindSpore 集成能力

#### FR-8 MindSpore 模型封装
系统必须使用 MindSpore 对求解器中的训练单元或量子层进行封装，使其具备标准化参数管理与训练入口。

#### FR-9 训练接口标准化
系统必须提供统一的训练/优化接口，使调用方能够通过一致的配置方式执行：
1. 单点 VQE；
2. 参数扫描 VQE；
3. 可选的评估过程。

#### FR-10 后续模型扩展接口
系统应保留将量子层接入更复杂 MindSpore 模型的能力，包括但不限于：
1. 经典网络 + 量子层的组合；
2. 量子参数与经典参数的统一管理；
3. 后续实验中替换优化器或损失定义。

### 7.3 数据输入输出能力

#### FR-11 参数化配置输入
系统必须支持通过明确配置指定：
1. qubit 数；
2. 电路层数/深度；
3. 哈密顿量类型；
4. 控制参数范围；
5. 优化器类型与超参数；
6. 随机种子；
7. 输出路径。

#### FR-12 结构化结果输出
系统必须输出结构化结果，至少包括：
1. 每次优化的目标参数值；
2. 最优能量；
3. 最优参数向量；
4. 迭代次数；
5. 收敛状态；
6. 必要的元信息（系统规模、深度、种子、时间戳、配置摘要）。

#### FR-13 与现有数据流程兼容
系统应尽可能保证导出结果可被现有 Python 数据处理流程复用，尤其要保证：
1. 参数顺序稳定；
2. 标签字段清晰；
3. 导出格式与当前 CSV 消费方式兼容或可低成本适配。

### 7.4 可用性与工程化能力

#### FR-14 可执行入口
系统必须提供一个清晰的 Python 入口，用于运行新增的 MindQuantum VQE 功能。该入口可以是脚本、模块入口或 notebook，但应优先选择可重复执行的工程化形式。

#### FR-15 日志与过程反馈
系统必须在执行过程中输出必要日志，至少覆盖：
1. 当前配置摘要；
2. 当前扫描点/训练轮次；
3. 当前能量或损失；
4. 收敛或失败原因。

#### FR-16 错误处理
系统必须对以下错误提供可识别反馈：
1. MindQuantum / MindSpore 未安装或版本不兼容；
2. 配置非法；
3. 哈密顿量/线路构建失败；
4. 优化未收敛；
5. 输出路径不可写。

## 8. 非功能需求

### 8.1 正确性

#### NFR-1
在小规模系统上，新实现的能量结果必须能够与以下任一基准进行交叉验证：
1. 当前 Julia/Yao 实现；
2. 解析结果；
3. 独立数值对角化结果。

#### NFR-2
参数顺序、qubit 索引和哈密顿量项的符号语义必须保持一致，避免“能运行但结果错误”的静默偏差。

### 8.2 可维护性

#### NFR-3
新增代码必须模块化组织，至少在逻辑上分离：
1. 线路构建；
2. 哈密顿量构建；
3. 求解循环；
4. 配置与结果导出；
5. MindSpore 封装层。

#### NFR-4
新增实现不得与现有 Julia / PyTorch 代码强耦合到无法单独运行。

### 8.3 可扩展性

#### NFR-5
新增实现应支持后续扩展更多哈密顿量、更多 ansatz 和更多优化策略，而不需要重写核心控制流。

#### NFR-6
设计上应允许未来逐步把更多经典模型迁移到 MindSpore，而非一次性大迁移。

### 8.4 性能与资源

#### NFR-7
默认实现应面向开发与研究用途，优先保证正确性与可验证性，其次再优化性能。

#### NFR-8
在本机环境中应优先假设 CPU 可用，不将 GPU 作为前置依赖。

### 8.5 可复现性

#### NFR-9
系统必须支持设置随机种子，并记录关键运行配置，保证实验可复现。

#### NFR-10
相同配置在相同软件环境下重复运行时，结果偏差应保持在可接受范围内，并可通过日志或元数据回溯。

## 9. 约束与假设

### 9.1 技术约束

1. 量子计算库固定为用户指定版本线的 **MindQuantum**；
2. 模型构建框架固定为 **MindSpore**；
3. 必须考虑当前仓库已有 Julia/Yao 与 PyTorch 代码并存的现实，不得假设仓库为空白重建；
4. 必须优先兼容当前仓库已有的自旋模型问题设定，而不是转向化学分子专用工作流。

### 9.2 环境约束

1. 需考虑 Python、MindQuantum、MindSpore 的版本兼容关系；
2. 需考虑 macOS 本地开发环境可用性；
3. 需考虑 MindQuantum 某些能力依赖 MindSpore 运行模式与版本匹配。

### 9.3 迁移假设

1. 现有 Julia 代码可作为算法语义参考实现；
2. 现有 Python 数据读取与训练流程将在短期内继续存在；
3. 新增能力优先以“并行新链路”的方式落地，而不是“原地替换所有旧链路”。

## 10. 兼容性要求

### 10.1 与现有代码兼容

1. 新功能不得破坏现有 Julia VQE 数据生成流程；
2. 新功能不得破坏现有 PyTorch 模型与 notebook 的基本使用；
3. 若新增输出目录或文件结构，应避免与现有数据目录命名冲突。

### 10.2 与现有数据兼容

1. 导出数据格式应尽可能复用现有 CSV 组织方式；
2. 如无法完全复用，必须提供清晰、一致、稳定的新格式；
3. 参数向量的排列顺序必须固定，便于下游消费。

## 11. 风险需求

以下风险必须在设计阶段被显式处理：

1. **版本风险**：MindQuantum / MindSpore / Python 版本不匹配；
2. **平台风险**：macOS 环境下的安装与运行限制；
3. **语义风险**：Julia 1-based 索引迁移到 Python 0-based 时出现 off-by-one；
4. **端序风险**：不同框架下 qubit ordering / Pauli 项语义不一致；
5. **优化风险**：warm start 未保留导致扫描任务收敛性能下降；
6. **接口风险**：MindSpore 封装与纯 VQE 优化流程耦合不当；
7. **兼容风险**：导出结果与现有下游训练流程不兼容。

## 12. 验收标准

### 12.1 功能验收

以下条件全部满足时，功能需求视为达成：

1. 可以在 Python 中运行一次完整 MindQuantum VQE 求解；
2. 可以对至少一种现有自旋哈密顿量完成求解；
3. 可以完成至少一种现有层状 ansatz 的构建与优化；
4. 可以完成参数扫描并支持 warm start；
5. 可以导出结构化结果；
6. 可以通过 MindSpore 封装执行核心训练/优化单元；
7. 对关键错误场景具备可读反馈。

### 12.2 数值验收

1. 在小规模问题上，VQE 最优能量应与参考结果保持一致或误差在可接受范围内；
2. 哈密顿量项数、参数总数和输出维度应与配置一致；
3. 参数扫描过程中的输出点数、标签和结果顺序应正确。

### 12.3 工程验收

1. 新增能力具备明确入口；
2. 配置、日志、输出结构清晰；
3. 新旧链路可以并存运行；
4. 至少具备最小可验证样例或测试策略。

## 13. 成功指标

本次需求完成后，应能够衡量以下成功指标：

1. 开发者无需依赖 Julia 环境即可运行核心 VQE 实验；
2. 新增 Python VQE 链路能在小规模问题上稳定复现基准；
3. 导出结果可用于现有或后续模型实验；
4. 新能力形成后续 MindSpore 混合模型扩展基础，而无需推倒重来。

## 14. 待确认问题

以下问题需在设计前由审查者确认或在设计中作出决策：

1. 本阶段是否要求新增实现完全对齐现有 `getDataset.jl` 的输出目录结构；
2. 本阶段 MindSpore 封装是否以 `MQAnsatzOnlyLayer` 风格为主，还是允许先使用更轻量的求解封装；
3. 本阶段是否要求同步提供 notebook 示例；
4. 是否要求在第一版中覆盖多种 ansatz，还是先实现最小可用的一种主线路模板；
5. 是否要求同步规划 PyTorch 模型向 MindSpore 的迁移路线。

## 15. 需求冻结原则

1. 本文档审查通过后，后续设计文档必须严格以本文档为依据；
2. 若设计阶段发现需求冲突或缺失，必须先回到本文档补充或修订；
3. 任务拆分文档不得引入超出本文档范围的新目标，除非获得显式批准。
